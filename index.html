School ERP SaaS Platform

Complete Software Requirement Specification (SRS), Functional & Technical Design, Database Design, API Specification, and System Architecture for a multi-tenant, multi-organization, multi-branch enterprise School ERP.

Scale target: 1,000+ Organizations · 10,000+ Branches · 1,000,000+ Students

Angular + Material Node.js / Express PostgreSQL Redis Cache Bull Queue AWS S3 JWT + RBAC Swagger / OpenAPI Multi-Tenant SaaS Microservice-Ready

Document version 1.0 · Status: Design Baseline · Audience: Architecture, Product, Engineering, QA, DevOps

1 Project Overview

A cloud-native, multi-tenant SaaS platform that lets a single deployment serve thousands of independent schools (organizations), each running multiple physical branches, with strict data isolation, role-based access, and a unified billing/operations layer.

1.1 Vision

Deliver an all-in-one digital backbone for K-12 and higher-secondary institutions covering academics, administration, finance, operations and parent/student engagement — sold as a subscription product to school chains and individual schools.

1.2 Objectives

  • Single platform, many tenants: one codebase, logically isolated data per organization.
  • Operational efficiency: automate admissions, attendance, fees, exams, payroll, transport and hostel operations.
  • Engagement: real-time communication with parents and students via app, SMS, email and push.
  • Compliance & audit: immutable audit trail of every privileged action.
  • Scale & reliability: horizontally scalable, ≥99.9% availability.

1.3 Tenancy Model

PLATFORM (Super Admin) └── ORGANIZATION (Tenant — e.g. "Delhi Public School Group") org_id ├── BRANCH (Campus — e.g. "DPS Whitefield") branch_id │ ├── Academic Years → Classes → Sections │ ├── Students / Parents / Employees / Teachers │ └── Fees / Exams / Transport / Hostel / Library / Inventory └── BRANCH (Campus — "DPS Indiranagar")
Isolation strategy: Shared database, shared schema with a mandatory org_id (and where relevant branch_id) discriminator column on every tenant table, enforced by a Row-Level-Security policy + an application-tier tenant guard. Premium / enterprise tenants can be promoted to a dedicated schema or dedicated DB without code change (repository abstraction).

FieldDetail
ActorParent (primary), Payment Gateway (secondary), Accountant (observer)
Pre-conditionsParent authenticated; student has outstanding dues; gateway configured.

13 API Specification

REST, JSON, versioned under /api/v1. All non-auth endpoints require Authorization: Bearer <access>. Tenant context (org_id, branch_id) is derived from the token, never the body. Standard envelope, pagination, and error format below.

14 Database Design (PostgreSQL)

PostgreSQL note: Postgres does not support MySQL-style inline COMMENT '...' in CREATE TABLE. Column intent is documented with inline -- comments and formal COMMENT ON COLUMN statements (shown once below). Every tenant table carries org_id (+ branch_id where applicable) and standard audit columns.

14.1 Conventions

  • PKs: BIGSERIAL (or BIGINT GENERATED ALWAYS AS IDENTITY).
  • Audit columns on every table: created_by, created_at, updated_by, updated_at, deleted_at (soft delete). Log tables are append-only — no updated_*.
  • Tenant discriminator: org_id NOT NULL + RLS policy org_id = current_setting('app.org_id')::bigint.
  • High-volume tables (attendance, audit_logs, api_logs, notifications) are range-partitioned by month.
  • Money as NUMERIC(14,2); never floats.

14.2 Core Tenancy & Identity

CREATE TABLE organizations (
  id              BIGSERIAL PRIMARY KEY,            -- Tenant primary key
  name            VARCHAR(150) NOT NULL,            -- School/group name
  code            VARCHAR(40)  NOT NULL UNIQUE,     -- Short unique tenant code
  subdomain       VARCHAR(60)  UNIQUE,              -- White-label subdomain
  plan            VARCHAR(20)  NOT NULL DEFAULT 'STARTER', -- STARTER|PRO|ENTERPRISE
  status          VARCHAR(20)  NOT NULL DEFAULT 'ACTIVE',  -- ACTIVE|SUSPENDED|TRIAL
  max_students    INT          NOT NULL DEFAULT 1000,     -- Plan quota
  region          VARCHAR(20)  NOT NULL DEFAULT 'ap-south-1', -- Data residency
  settings        JSONB        NOT NULL DEFAULT '{}',      -- Branding/feature flags
  created_by      BIGINT,
  created_at      TIMESTAMPTZ  NOT NULL DEFAULT now(),
  updated_by      BIGINT,
  updated_at      TIMESTAMPTZ,
  deleted_at      TIMESTAMPTZ
);
COMMENT ON TABLE organizations IS 'Top-level SaaS tenant';
COMMENT ON COLUMN organizations.settings IS 'JSONB: logo, theme, enabled modules';

CREATE TABLE branches (
  id              BIGSERIAL PRIMARY KEY,
  org_id          BIGINT NOT NULL REFERENCES organizations(id),
  name            VARCHAR(150) NOT NULL,
  code            VARCHAR(40)  NOT NULL,
  address         TEXT,
  city            VARCHAR(80),
  timezone        VARCHAR(40) NOT NULL DEFAULT 'Asia/Kolkata',
  status          VARCHAR(20) NOT NULL DEFAULT 'ACTIVE',
  created_by BIGINT, created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_by BIGINT, updated_at TIMESTAMPTZ, deleted_at TIMESTAMPTZ,
  UNIQUE(org_id, code)
);
CREATE INDEX idx_branches_org ON branches(org_id) WHERE deleted_at IS NULL;

CREATE TABLE users (
  id              BIGSERIAL PRIMARY KEY,
  org_id          BIGINT NOT NULL REFERENCES organizations(id),
  branch_id       BIGINT REFERENCES branches(id),     -- NULL for org-level users
  username        VARCHAR(80),
  email           VARCHAR(160) NOT NULL,
  phone           VARCHAR(20),
  password_hash   VARCHAR(255) NOT NULL,           -- bcrypt/argon2
  status          VARCHAR(20) NOT NULL DEFAULT 'ACTIVE',
  mfa_enabled     BOOLEAN NOT NULL DEFAULT false,
  last_login_at   TIMESTAMPTZ,
  created_by BIGINT, created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_by BIGINT, updated_at TIMESTAMPTZ, deleted_at TIMESTAMPTZ,
  UNIQUE(org_id, email)
);
CREATE INDEX idx_users_org_branch ON users(org_id, branch_id);

CREATE TABLE roles (
  id          BIGSERIAL PRIMARY KEY,
  org_id      BIGINT REFERENCES organizations(id),    -- NULL = system role
  name        VARCHAR(60) NOT NULL,
  slug        VARCHAR(60) NOT NULL,             -- e.g. CLASS_TEACHER
  is_system   BOOLEAN NOT NULL DEFAULT false,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE(org_id, slug)
);

CREATE TABLE permissions (
  id          BIGSERIAL PRIMARY KEY,
  code        VARCHAR(80) NOT NULL UNIQUE,      -- module.action e.g. fee.collect
  module      VARCHAR(40) NOT NULL,
  description VARCHAR(160)
);

17.2 Authorize Middleware (illustrative)

const authorize = (permission) => (req, res, next) => {
  const { permissions, orgId } = req.user;          // from JWT + redis cache
  if (!permissions.includes(permission))
    return next(new ForbiddenError('FORBIDDEN'));
  // tenant scope already pinned via SET app.org_id; deny cross-tenant id
  next();
};

18 Security Architecture

ControlImplementation
AuthenticationJWT access (15 min, RS256) + refresh (7 days, rotating, stored hashed in Redis with jti).
Refresh & RevocationRefresh rotation, reuse-detection (revoke family), logout invalidates jti, device list.
Passwordbcrypt (cost 12) / argon2id, breach-check, complexity policy, history, lockout after N fails.
MFATOTP/Email OTP enforced for admin roles.
AuthorizationRBAC permission checks per endpoint + Postgres RLS tenant isolation.
Rate LimitingRedis token-bucket: global, per-IP, per-user; stricter on /auth.
SQL InjectionParameterized queries / ORM bindings only; no string concat.

19.1 Environments & Scaling

  • Environments: dev → staging → production, infra-as-code (Terraform), per-region.
  • Autoscaling: HPA on CPU/req-latency for API & worker pods; DB read replicas for reporting.
  • Microservice-ready: modular monolith now; bounded contexts (Auth, Academics, Finance, Comms) extractable to services behind the same gateway.
  • Caching: Redis for permissions, hot masters, dashboards; CDN for static + cacheable GETs.
  • DR: automated daily snapshots + PITR; cross-region backup replication; RPO ≤ 5 min, RTO ≤ 30 min.
